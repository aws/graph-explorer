```bash
# Pull and run Graph Explorer
docker pull public.ecr.aws/neptune/graph-explorer
docker run -p 443:443 \
  -e GRAPH_CONNECTION_URL=https://your-neptune-endpoint:8182 \
  public.ecr.aws/neptune/graph-explorer
```

```typescript
interface ConnectionConfig {
  url: string;
  queryEngine: "gremlin" | "openCypher" | "sparql";
  awsAuthEnabled?: boolean;
  awsRegion?: string;
}

function createConnection(config: ConnectionConfig): Connection {
  const { url, queryEngine, awsAuthEnabled = false } = config;
  return new Connection({ url, queryEngine, awsAuthEnabled });
}
```

```json
{
  "connection": {
    "url": "https://your-neptune-endpoint:8182",

  }
}
```

## Long Content for Scroll Testing

The following paragraph is intentionally verbose to test how long-form content renders and whether line-length constraints work properly at various viewport widths.

Graph databases model data as nodes (vertices) and edges (relationships), making them ideal for use cases where the connections between entities are as important as the entities themselves. Social networks, fraud detection, recommendation engines, knowledge graphs, and network topology are all natural fits for graph databases. Amazon Neptune supports both property graph (via Gremlin and openCypher) and RDF graph (via SPARQL) models within the same service, allowing teams to choose the query language that best fits their use case without managing separate database engines.
